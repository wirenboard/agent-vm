# claude-vm-shim

**Status: working prototype.** Each session opened from `claude.ai/code` or
the Claude mobile app via `claude remote-control` lands inside a fresh
[`agent-vm`](https://github.com/wirenboard/agent-vm) sandbox instead of
running on the host. Verified: cloud session attaches, the in-VM claude
calls api.anthropic.com (auth via the `/agent-vm-state` mount), and you can
interact with it from the mobile app while the host stays untouched.

This branch is the **remote-control–only** flavour. The sibling branch
[`claude-vm-shim`](../../tree/claude-vm-shim) additionally handles the
`claude --bg` flow (background CLI sessions), which uses a separate
per-session spawn protocol (`--bg-spare` + UDS handshake). Use the rc-only
target if you don't need `--bg`.

## Why this exists

When you run `claude remote-control` and connect from the mobile app, the
local daemon spawns one process per cloud-opened session:

```
claude --print --sdk-url https://api.anthropic.com/v1/code/sessions/cse_… \
       --session-id cse_… --input-format stream-json \
       --output-format stream-json --replay-user-messages
```

That process talks JSON-stream over its stdio: input from the cloud
(`stream-json` on stdin), output back (`stream-json` on stdout), and the
session itself is routed through the supplied `--sdk-url`.

This project intercepts the per-session exec via `LD_PRELOAD`, reroutes it
into an `agent-vm`, and pipes stdio through. The daemon thinks it's talking
to a local subprocess; underneath, it's a subprocess inside a VM.

## Architecture

```
   ┌─────────────────── host ─────────────────────┐    ┌── agent-vm guest ──┐
   │ claude remote-control                         │    │                    │
   │   │                                           │    │                    │
   │   │ posix_spawn(claude --print --sdk-url …)   │    │                    │
   │   ▼                                           │    │                    │
   │ libclaude_vm_shim.so  ── rewrites argv ─┐     │    │                    │
   │                                          │    │    │                    │
   │ claude-vm-dispatcher  ◀──────────────────┘    │    │                    │
   │   │  stdio inherited                          │    │                    │
   │   │                                           │    │                    │
   │   └──spawn──▶ agent-vm claude --no-git  ──────┼────┤  claude --print    │
   │                       stdio piped through     │    │    --sdk-url …     │
   │                                               │    │                    │
   │   propagate exit code back ◀──────────────────┼────┤  (cloud session    │
   │                                               │    │   runs here)       │
   └───────────────────────────────────────────────┘    └────────────────────┘
```

Two executable artifacts in `/opt/claude-vm-shim/`:

- `lib/libclaude_vm_shim.so` — `LD_PRELOAD` cdylib that intercepts
  `execve`/`execvp`/`posix_spawn`/`posix_spawnp`. When argv matches
  `[<claude>, "--print", …, "--sdk-url", …]`, it rewrites the call to
  invoke the dispatcher with the original argv as its arguments.
- `bin/claude-vm-dispatcher` — handles a single cloud-session invocation:
  resolves the `agent-vm` binary, boots `agent-vm claude --no-git -- …`
  with inherited stdio, waits, propagates the exit code. Falls back to a
  local `claude` exec on failure or when `CLAUDE_VM_SHIM_PASSTHROUGH=1`.

A `bin/claude-shimmed` wrapper script sets `LD_PRELOAD` and execs the real
`claude` binary, so the daemon and every subprocess it spawns inherit the
interception. The wrapper script is generated by `install.sh` with the
real-claude path baked in.

```
claude-vm-shim/
├── Cargo.toml             # workspace (shim + dispatcher)
├── shim/                  # libclaude_vm_shim.so   (cdylib, LD_PRELOAD)
```

No changes outside this directory.

## Building

Prerequisites:

- `gcc` + `libc6-dev`
- `npm install -g @wirenboard/agent-vm` (provides `msb` + `libkrunfw`)
  and `agent-vm setup --no-verify` to pull the OCI image
- The repo's vendored Rust toolchain (use `PATH=$REPO/.agent-vm-rust/cargo/bin:$PATH`
  and `RUSTUP_HOME`/`CARGO_HOME` pointing at `$REPO/.agent-vm-rust/`)

Build:

```sh
cd claude-vm-shim
cargo build --release
./install.sh
```

After install:

```
/opt/claude-vm-shim/
├── bin/
│   ├── claude-shimmed            # the wrapper exposed at ~/.local/bin/
│   ├── claude-vm-dispatcher      # host-side per-session entry
│   └── claude-wrapper.sh         # sets LD_PRELOAD, execs real claude
└── lib/
    └── libclaude_vm_shim.so      # LD_PRELOAD library
```

## Using it

```sh
# One-time consent (claude remote-control normally asks "Enable Remote
# Control? (y/n)" on first run; we set the persistent flag here).
python3 -c 'import json,os; p=os.path.expanduser("~/.claude.json"); \
            d=json.load(open(p)); d["hasUsedRemoteControl"]=True; \
            json.dump(d,open(p,"w"))'

# Trust the workspace you intend to run in (rc refuses untrusted ones):
python3 -c 'import json,os,sys; p=os.path.expanduser("~/.claude.json"); \
            d=json.load(open(p)); d.setdefault("projects",{}).setdefault(sys.argv[1],{})["hasTrustDialogAccepted"]=True; \
            json.dump(d,open(p,"w"))' /tmp/test-vm

# Run remote-control via the shim:
cd /tmp/test-vm
CLAUDE_VM_SHIM_LOG=/tmp/cvs.log claude-shimmed remote-control --spawn=same-dir
```

When the cloud opens a session you'll see something like:

```
✔︎ Connected · test-vm · HEAD
    Capacity: 1/32 · New sessions will be created in the current directory
    https://claude.ai/code/session_…  Attached
```

In another shell, confirm the session is inside a VM:

```sh
tail -F /tmp/claude-vm-dispatcher.log
# expect: cloud session begin: 10 forwarded args
#         spawning … agent-vm claude --no-git -- […]
#         agent-vm child pid=…

ps --forest -ef | grep -E 'remote-control|claude-vm|msb sand' | grep -v grep
# you should see:
#   remote-control
#    └── claude-vm-dispatcher --print --sdk-url …
#         └── agent-vm claude --no-git -- …
#              └── msb sandbox agent-vm-… (the VM)
```

## Env knobs

| Env var                          | Effect |
|----------------------------------|--------|
| `CLAUDE_VM_SHIM_LOG`             | Path; shim + dispatcher append logs here in addition to `/tmp/claude-vm-dispatcher.log`. |
| `CLAUDE_VM_SHIM_PASSTHROUGH=1`   | Skip the VM dispatch; exec a host claude verbatim. For A/B comparisons. |
| `CLAUDE_VM_SHIM_DISPATCHER`      | Override the dispatcher path the shim invokes. Set by the wrapper script. |
| `CLAUDE_VM_SHIM_SO`              | Path to `libclaude_vm_shim.so` (used by the wrapper). |
| `CLAUDE_VM_SHIM_REAL_CLAUDE`     | Path to the real claude binary (used by the wrapper). |
| `CLAUDE_VM_SHIM_AGENT_VM_BIN`    | Override the `agent-vm` binary path the dispatcher uses. |
| `CLAUDE_VM_SHIM_WORKDIR_ROOT`    | Per-session VM workdir root (defaults to `/tmp/claude-vm-shim-work`). |
| `CLAUDE_VM_SHIM_MSB_PATH`        | `MSB_PATH` forwarded to the agent-vm child. Defaults to the npm-bundled `msb`. |

## How interception works (LD_PRELOAD with Bun)

The Claude Code binary is a Bun-compiled standalone. Bun's spawn primitives
call libc's `execve`/`posix_spawn` through normal dynamic linkage (PLT/GOT),
not via `dlopen("libc.so.6")` + `dlsym(handle, "execve")`. That means
`LD_PRELOAD` overrides do take precedence. Confirmed empirically — see
`tail -F /tmp/claude-vm-dispatcher.log` after running `claude remote-control`.

The shim matches on `argv[1] == "--print" && argv.contains("--sdk-url")`
rather than on argv[0]'s path, because the daemon spawns with
`pinToCurrentBinary: true` and the path is
`/root/.local/share/claude/versions/<v>` (it can change on upgrade). The
`--sdk-url` requirement disambiguates from plain `claude --print "hi"`.

The dispatcher unsets `CLAUDE_VM_SHIM_DISPATCHER` before any inner exec,
so the shim short-circuits and never recurses inside the VM.

## What works

- ✅ Per-session VM boot (one fresh agent-vm per cloud session).
- ✅ Cloud session attaches; rc parent shows `Capacity: N/32 · …Attached`.
- ✅ In-VM claude calls api.anthropic.com via the standard public-egress
  path (no policy override needed; cloud-session traffic is "public").
- ✅ Auth/credentials via the `/agent-vm-state` mount (the same way
  `agent-vm claude` works for an interactive CLI session).
- ✅ stream-json protocol flows through inherited stdin/stdout/stderr.

## Known gaps

- **`--spawn=worktree`.** Untested. The daemon's `WorktreeCreate` hook
  would fire before our shim sees the per-session exec; behavior with the
  hook + VM combination is unverified.
- **Reattach across rc restarts.** The dispatcher creates a fresh VM per
  cloud session; if the rc parent crashes and the cloud reattaches the
  same session ID, we boot a *new* VM rather than reconnecting to the
  existing one. Acceptable for now; would need session-id → vm-state
  tracking to do better.
- **OAuth refresh across long sessions.** The in-VM claude reads OAuth
  from `/agent-vm-state/claude/.credentials.json` via the state mount.
  Refresh works through the existing intercept-hook + secrets layer for
  the standard agent-vm CLI flow; haven't exercised with an expired
  token during a live rc session.
